# Notion 데이터베이스 구조

## 1. 최종 속성 구성

| 속성명 | 타입 | 설명 |
| --- | --- | --- |
| 제목 | Title | RSS에서 수집한 기사 제목 |
| 요약문 | Rich text | Gemini가 생성한 3줄 이내의 한국어 요약문 |
| 원문 링크 | URL | 기사 원문 링크이며, 동일 기사 중복 저장을 방지하는 기준 값 |
| 발행일시 | Date | RSS의 `pubdate` 값 |
| 상태 | Select | 정상 뉴스와 오류 기록을 구분하는 `성공` 또는 `오류` 값 |
| 오류 로그 | Rich text | 최종 실패 시 확인된 오류 메시지와 상세 내용 |
| 오류 ID | Rich text | Make Incomplete execution 항목의 중복 기록을 방지하기 위한 고유 값 |

## 2. 속성별 운영 기준

### 제목

RSS의 기사 제목을 저장한다. Notion 페이지의 대표 제목으로 사용되며, 사용자가 데이터베이스에서 기사를 빠르게 식별할 수 있게 한다.

### 요약문

Gemini의 출력 결과를 그대로 저장한다. 요약은 반드시 3줄 이내이며 각 줄은 `- `로 시작한다. 오류 기록 항목에서는 비워 둘 수 있다.

### 원문 링크

RSS 기사 URL을 저장하며 **중복 방지 키**로 사용한다. 메인 시나리오는 본문 수집과 AI 요약 전에 이 속성을 검색하여 동일 링크가 존재하는 경우 처리를 종료한다. 이로써 중복 저장뿐 아니라 불필요한 HTTP·Gemini 호출도 방지한다.

### 발행일시

RSS의 `pubdate`를 Notion Date 형식에 맞게 저장한다. 기사 게시 시점을 기준으로 정렬하거나 기간별로 조회할 때 사용한다.

### 상태

정상 저장과 오류 기록을 구분한다. 메인 워크플로우에서 정상 처리된 기사는 `성공`, 오류 모니터링 시나리오가 기록한 실패 항목은 `오류`로 저장한다.

### 오류 로그

Retry 후에도 실패한 실행의 오류 메시지와 필요한 진단 정보를 기록한다. 민감한 인증 정보, Authorization 헤더, 개인 정보는 저장하지 않는다.

### 오류 ID

Make Incomplete execution을 식별할 수 있는 고유 값을 저장한다. 오류 모니터링 시나리오는 이 값을 먼저 검색하여 동일 오류가 Notion에 반복 저장되는 것을 방지한다.

## 3. 최종 구조 결정 사항

초기 검토 과정에서 오류가 발생한 위치를 별도 속성으로 저장하는 `오류 단계` 항목을 고려했으나, 최종 구조에서는 제외하였다. 오류 발생 모듈과 실행 흐름은 Make의 History 및 Incomplete executions에서 확인할 수 있고, Notion의 `오류 로그`에도 필요한 상세 내용을 기록할 수 있기 때문이다. 속성을 단순화하여 데이터베이스 관리와 유지보수의 부담을 줄였다.

## 4. 정상·오류 데이터 구분

- 정상 기사: 제목, 요약문, 원문 링크, 발행일시를 저장하고 상태를 `성공`으로 설정한다.
- 최종 실패 기록: 상태를 `오류`로 설정하고 오류 로그와 오류 ID를 저장한다.
- 중복 뉴스는 원문 링크로, 중복 오류는 오류 ID로 각각 구분한다.

Notion 속성 매핑 화면과 실제 저장 결과는 `스크린샷/10_노션_저장_매핑/` 및 `스크린샷/11_노션_저장_결과/`에서 확인할 수 있다.
