# 단계별 역할과 연결 구조

## 1. RSS 수집 단계

- **사용 모듈:** RSS `Watch Feed Items`
- **입력값:** AI타임스 RSS 주소 `https://www.aitimes.com/rss/allArticle.xml`, `Asia/Seoul` 기준 매일 09:00 실행 스케줄
- **처리 내용:** 매일 09:00에 RSS 피드를 확인하여 새 기사 항목을 수집한다. 시연에서는 `All RSS feed items`, 운영·제출에서는 `From now on`을 시작 기준으로 사용한다.
- **출력값:** 기사 제목, 설명, 요약, 원문 링크, 발행일시
- **다음 단계와의 연결:** 제목·설명·요약을 로봇 키워드 필터에 전달하며, 링크와 발행일시는 중복 확인 및 최종 저장에 사용한다.

## 2. 로봇 키워드 필터링 단계

- **사용 모듈:** Make 연결 필터
- **입력값:** RSS의 `Title`, `Description`, `Summary`
- **처리 내용:** 다음 조건 중 하나라도 참이면 기사를 통과시킨다.

  ```text
  Title contains "로봇"
  OR
  Description contains "로봇"
  OR
  Summary contains "로봇"
  ```

- **출력값:** 로봇 주제와 관련된 RSS 기사 1건
- **다음 단계와의 연결:** 필터를 통과한 기사만 Notion 중복 확인 단계로 전달한다. 관련 없는 기사는 여기서 종료하여 후속 API 호출을 방지한다.

## 3. Notion 중복 확인 단계

- **사용 모듈:** Notion `Search Objects`
- **입력값:** RSS 기사 원문 링크
- **처리 내용:** Notion 데이터베이스의 `원문 링크` 속성에서 RSS 링크와 동일한 값을 검색한다. 원문 링크를 중복 방지 키로 사용한다.
- **출력값:** 동일 원문 링크를 가진 Notion 항목의 검색 결과와 결과 개수
- **다음 단계와의 연결:** 검색 결과 개수를 중복 없음 필터의 판단 값으로 전달한다.

## 4. 중복 없음 필터 단계

- **사용 모듈:** Make 연결 필터
- **입력값:** Notion 검색 결과 개수
- **처리 내용:** 검색 결과가 0건인 경우에만 신규 기사로 판단한다. 1건 이상이면 중복 기사이므로 실행을 종료한다.
- **출력값:** Notion에 저장되지 않은 신규 기사
- **다음 단계와의 연결:** 신규 기사만 HTTP 원문 수집 단계로 보낸다. 이 배치로 중복 기사에 대한 HTTP 요청과 Gemini 호출을 차단한다.

## 5. HTTP 원문 HTML 수집 단계

- **사용 모듈:** HTTP `Make a request`
- **입력값:** RSS 기사 원문 링크, 요청 방식 `GET`
- **처리 내용:** 인증 정보 없이 공개 기사 페이지에 요청을 보내 HTML 전체를 수집한다. RSS 설명문보다 충분한 본문을 확보하기 위한 단계이다.
- **출력값:** 기사 페이지의 HTML 응답 본문과 응답 상태
- **다음 단계와의 연결:** 정상 HTML 응답을 Text Parser에 전달한다. 네트워크 또는 서버 오류는 Retry 정책의 대상이 된다.

## 6. Text Parser 본문 문단 추출 단계

- **사용 모듈:** Text Parser `Match Pattern`
- **입력값:** HTTP 응답 HTML
- **처리 내용:** 다음 정규식으로 속성 유무와 관계없이 `<p>` 시작 태그부터 `</p>` 종료 태그까지의 문단을 추출한다.

  ```regex
  <p[^>]*>[\s\S]*?<\/p>
  ```

- **출력값:** 여러 개로 분리된 `<p>` 문단 결과
- **다음 단계와의 연결:** 추출된 모든 문단을 Text Aggregator에 전달한다.

## 7. Text Aggregator 본문 병합 단계

- **사용 모듈:** Tools `Text Aggregator`
- **입력값:** Text Parser가 추출한 여러 `<p>` 문단
- **처리 내용:** 문단 사이에 줄바꿈을 적용하여 하나의 기사 본문 문자열로 합친다. 문단별 개별 AI 호출을 방지하고 기사 단위 입력을 만든다.
- **출력값:** 하나로 병합된 HTML 형식의 기사 본문
- **다음 단계와의 연결:** 병합된 본문을 HTML to text 모듈에 전달한다.

## 8. HTML to text 본문 정제 단계

- **사용 모듈:** Text Parser `HTML to text`
- **입력값:** Text Aggregator가 병합한 본문
- **처리 내용:** `<p>` 등의 HTML 태그를 제거하고 HTML 엔티티를 일반 문자로 변환한다. AI 입력에서 마크업과 불필요한 표현을 줄인다.
- **출력값:** 사람이 읽을 수 있는 일반 텍스트 기사 본문
- **다음 단계와의 연결:** 정제된 본문을 기사 제목·링크와 함께 Gemini에 전달한다.

## 9. Gemini AI 요약 단계

- **사용 모듈:** Google Gemini AI `Generate a response`
- **입력값:** 기사 제목, 원문 링크, 정제된 기사 본문
- **처리 내용:** 다음 조건을 포함한 프롬프트로 한국어 요약문을 생성한다.

  ```text
  - 반드시 3줄 이내로 작성
  - 각 줄은 "- "로 시작
  - 기사에 없는 내용은 추측하지 않기
  - 로봇 기술 동향, 기업/시장 영향, 핵심 시사점을 중심으로 요약
  - 광고, 메뉴, 댓글, 관련기사, 기자 정보는 요약하지 않기
  - 요약문만 출력하고 "요약:" 같은 제목은 붙이지 않기
  ```

- **출력값:** 각 줄이 `- `로 시작하는 3줄 이내의 한국어 요약문
- **다음 단계와의 연결:** 요약문을 RSS 메타데이터와 함께 Notion 저장 단계로 전달한다. 기사 1건당 1회 호출을 원칙으로 하며 오류 재시도만 예외로 허용한다.

## 10. Notion 데이터베이스 저장 단계

- **사용 모듈:** Notion `Create Database Item`
- **입력값:** RSS 제목·원문 링크·발행일시, Gemini 요약문, 정상 처리 상태
- **처리 내용:** 데이터를 `제목`, `요약문`, `원문 링크`, `발행일시`, `상태` 속성에 맞게 매핑한다. 정상 처리 항목의 상태는 `성공`으로 저장한다.
- **출력값:** 생성된 Notion 데이터베이스 항목
- **다음 단계와의 연결:** 저장된 원문 링크는 이후 실행에서 중복 방지 키로 사용된다. 정상 처리 흐름은 이 단계에서 종료된다.

## 11. 오류 처리 및 모니터링 구조

- **사용 모듈:** Make Retry, Incomplete executions, 오류 모니터링 시나리오, Notion 검색·저장 모듈
- **입력값:** HTTP 또는 Gemini 단계의 오류 정보, 실행 ID 또는 오류 ID, 오류 메시지, `Asia/Seoul` 기준 매일 09:10 실행 스케줄
- **처리 내용:** 오류 발생 시 최대 2회 재시도한다. 계속 실패하면 실행을 Incomplete executions에 저장한다. 오류 모니터링 시나리오는 매일 09:10에 실패 실행을 조회한 뒤 오류 ID가 Notion에 이미 있는지 확인하고, 신규 오류만 `상태: 오류`, `오류 로그`, `오류 ID`로 저장한다.
- **출력값:** 재시도로 복구된 정상 실행 또는 Notion에 기록된 최종 실패 로그
- **다음 단계와의 연결:** 복구된 실행은 원래 흐름을 이어 가고, 최종 실패는 오류 로그 저장 후 운영자가 Make History와 Incomplete executions에서 원인을 확인한다.

## 12. 최종 연결 구조 요약

```text
RSS Watch Feed Items
→ 로봇 기사 필터
→ Notion Search Objects
→ 중복 없음 필터
→ HTTP Make a request
→ Text Parser Match Pattern
→ Text Aggregator
→ HTML to text
→ Gemini Generate a response
→ Notion Create Database Item
```

이 구조는 필터링과 중복 확인을 비용이 발생할 수 있는 본문 수집·AI 요약보다 먼저 수행한다. 이어서 문단 추출, 병합, 텍스트 정제를 순서대로 적용하여 Gemini가 기사 본문 중심의 입력을 받도록 구성한다. 최종 결과는 Notion에 구조화하여 저장하며, 오류 흐름은 메인 처리와 분리해 추적 가능성과 유지보수성을 높였다.
