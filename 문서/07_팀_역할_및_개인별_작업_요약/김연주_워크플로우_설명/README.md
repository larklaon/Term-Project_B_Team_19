## 1. 팀역할

모든 팀원이 동일한 주제를 바탕으로 각자 독립적인 뉴스 자동화 워크플로우를 설계하고 구현하였다.
이후 팀원들이 제작한 워크플로우를 함께 비교·분석하며 구조, 안정성, 구현 난이도, 유지보수성을 검토하였다.
각 워크플로우의 장점과 개선점을 공유하고, 가장 효율적이고 안정적인 기능만을 취합하여 하나의 최종 워크플로우를 완성하였다.

## 2. 주제 필터링 기준과 선택 이유

AI Times는 AI 전반을 다루는 기술 전문 뉴스 매체이기 때문에, 단순히 'AI' 를 기준으로 필터링하면
반도체, 정책, 투자, 서비스, 생성형 AI 등 매우 다양한 분야의 기사가 함께 수집되는 문제가 있었다.

따라서 보다 정확한 자동화와 일관된 데이터 수집을 위해서는 하나의 구체적인 기술 분야를 선정하는 것이 적합하다고 판단하였다.
그 결과 '로봇' 을 최종 주제로 선택하였으며, 원하는 분야의 기사만 안정적으로 선별할 수 있었다.

## 3. 에러처리 정책과 이유

1. 초기 설계

초기에는 AI 요약 과정에서 오류가 발생하면 2회 재시도 후 노션 Error Log에 기록하는 구조를 계획하였다.
구현 과정에서 Make의 Error Handler는 재시도는 가능하지만, 재시도 성공 후 다음 모듈로 이어서 실행하는 기능은 지원하지 않는다는 점을 확인하였다.

2.  최종 적용 방식

최종적으로는 별도의 노션 Error Log를 구축하지 않고, Make에서 기본 제공하는 History를 활용하는 방식으로 변경하였다.

3.  선택 이유

- Make의 실행 로그 기능이 안정적으로 제공된다.
- 오류 원인과 실행 흐름을 한곳에서 확인할 수 있다.
- 별도의 노션 로그 관리가 필요 없어 유지보수가 간편하다.
- 불필요한 노션 API 호출을 줄여 워크플로우를 더욱 효율적으로 운영할 수 있다.
