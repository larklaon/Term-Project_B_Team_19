# 팀 역할 및 개인별 작업 요약

## 1. 팀 작업 방식

모든 팀원이 동일한 주제를 바탕으로 각자 독립적인 뉴스 자동화 워크플로우를 설계하고 구현하였다. 이후 팀원들이 제작한 워크플로우를 함께 비교·분석하며 구조, 안정성, 구현 난이도, 유지보수성을 검토하였다.

각 워크플로우의 장점과 개선점을 공유한 뒤, 가장 효율적이고 안정적인 기능을 중심으로 최종 제출용 워크플로우를 선정하였다. 최종 워크플로우는 RSS 수집, 로봇 키워드 필터링, Notion 중복 확인, HTTP 원문 수집, 본문 정제, Gemini 요약, Notion 저장, 오류 처리 구조를 포함한다.

구현 과정에서는 RSS 수집, 주제 필터링, Gemini 요약, Notion 저장, 중복 방지, 오류 처리 방식을 공통 검토 항목으로 삼았다. 실제 Make 모듈의 기능과 제약을 확인한 뒤 논의를 통해 최종 구조를 보완했으며, 선정된 워크플로우를 기준으로 단계별 동작과 정책을 문서화하였다.

## 2. 개인별 자료 구성

각 팀원 폴더에는 다음 자료를 정리한다.

- 개인별 워크플로우 설명 문서
- Make 워크플로우 스크린샷
- Notion 저장 결과 스크린샷

각 `이름_워크플로우_설명` 폴더 바로 아래에 `README.md`와 이미지 2개를 함께 배치하였다. GitHub에서 개인 폴더를 열면 설명 문서가 자동으로 표시되며, 별도의 단일 이미지용 하위 폴더 없이 한곳에서 개인별 자료를 확인할 수 있다.

## 4. 최종 워크플로우 선정 방식

최종 제출용 워크플로우는 팀원들이 각각 구현한 자동화 흐름을 비교한 뒤 선정하였다. 일부 워크플로우는 추가 기능이나 확장 기능을 포함하고 있었지만, 과제 요구사항 충족 여부, 설명 가능성, 유지보수성, 안정성을 기준으로 최종 구조를 결정하였다.

최종적으로는 Make 기반의 RSS 뉴스 수집, 로봇 키워드 필터링, Notion 중복 확인, HTTP 본문 수집, 본문 정제, Gemini 요약, Notion 저장 구조를 제출용 워크플로우로 사용하였다.

개별 구현 결과를 비교함으로써 단순히 기능이 동작하는지를 넘어 중복 방지 위치, AI 호출 횟수, 본문 정제 방식, 오류 복구 가능성을 함께 검토할 수 있었다. 최종 구조는 과제의 필수 요구사항을 충족하면서도 팀원 모두가 구조와 선택 이유를 설명할 수 있도록 구성하였다.

팀 문서와 스크린샷에는 API 키, 인증 토큰, 개인 이메일, 비공개 링크를 포함하지 않는다.
