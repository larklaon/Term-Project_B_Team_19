# 개인 워크플로우 설명 - 송영준

## 1. 개요

이 문서는 개인이 설계하고 구현한 Make 기반 뉴스 자동화 워크플로우의 단계별 역할과 연결 구조를 설명하기 위한 문서이다.

본 워크플로우는 AI타임스 RSS 피드에서 `로봇` 관련 뉴스를 자동으로 수집하고, Notion 데이터베이스에서 원문 링크 기준으로 중복 여부를 확인한 뒤, 중복이 아닌 기사만 원문 본문을 수집하여 Gemini로 요약하고 Notion에 저장하는 구조이다.

이 문서는 별도의 오류 모니터링 시나리오를 추가하기 전의 메인 워크플로우를 기준으로 작성하였다. 따라서 오류 처리는 별도 시나리오에서 Notion 오류 로그로 옮기는 방식이 아니라, Make의 Retry 기능과 History, Incomplete executions를 활용하는 단계까지를 기준으로 한다.

---

## 2. 사용 도구

- Make.com
- RSS
- HTTP
- Text Parser
- Text Aggregator
- Google Gemini AI
- Notion

---

## 3. 워크플로우 전체 구조

전체 워크플로우는 다음 순서로 구성된다.

```text
RSS 수집
→ 로봇 키워드 필터링
→ Notion 원문 링크 중복 확인
→ 중복 없음 필터
→ HTTP 원문 HTML 수집
→ Text Parser로 <p> 태그 본문 추출
→ Text Aggregator로 본문 병합
→ HTML to text로 본문 정제
→ Gemini AI 요약
→ Notion 데이터베이스 저장
```

Make 모듈 기준 연결 구조는 다음과 같다.

```text
RSS - Watch RSS feed items
→ Filter - 로봇 기사 필터
→ Notion - Search Objects
→ Filter - 중복 없음 필터
→ HTTP - Make a request
→ Text Parser - Match pattern
→ Tools - Text aggregator
→ Text Parser - HTML to text
→ Google Gemini AI - Generate a response
→ Notion - Create a Database Item
```

## 4. 단계별 역할과 연결 구조

### 4-1. RSS 수집 단계

- 사용 모듈: `RSS - Watch RSS feed items`
- 입력값: AI타임스 전체 기사 RSS 피드
- RSS 주소:

```text
https://www.aitimes.com/rss/allArticle.xml
```

- 처리 내용:
  - AI타임스 RSS 피드에서 최신 기사 정보를 가져온다.
  - 기사 제목, 설명, 요약, 원문 링크, 발행일시 등의 값을 수집한다.
  - 수집한 RSS 항목은 이후 필터링 단계의 입력값으로 사용된다.

- 출력값:
  - 기사 제목
  - 기사 설명 또는 요약
  - 원문 링크
  - 발행일시
  - RSS 원본 필드

- 다음 단계와의 연결:
  - 수집된 RSS 데이터는 `로봇` 키워드 필터로 전달된다.
  - 제목, 설명, 요약 필드를 기준으로 프로젝트 주제와 관련 있는 기사만 선별한다.

---

### 4-2. 로봇 키워드 필터링 단계

- 사용 모듈: Make 필터
- 입력값:
  - RSS Title
  - RSS Description
  - RSS Summary

- 처리 내용:
  - RSS로 수집된 기사 중 `로봇` 키워드가 포함된 기사만 통과시킨다.
  - AI타임스는 AI 전반을 다루는 기술 뉴스 매체이기 때문에, 단순히 `AI` 키워드로 필터링하면 반도체, 정책, 투자, 생성형 AI 등 다양한 주제의 기사가 함께 수집될 수 있다.
  - 따라서 더 구체적인 기술 분야인 `로봇`을 기준으로 필터링하여 수집 데이터의 일관성을 높였다.

- 필터 조건:

```text
Title contains "로봇"
OR
Description contains "로봇"
OR
Summary contains "로봇"
```

- 출력값:
  - 로봇 키워드가 포함된 기사 데이터

- 다음 단계와의 연결:
  - 필터를 통과한 기사만 Notion 중복 확인 단계로 전달된다.
  - 필터를 통과하지 못한 기사는 이후 단계로 진행하지 않는다.

---

### 4-3. Notion 중복 확인 단계

- 사용 모듈: `Notion - Search Objects`
- 입력값:
  - RSS 원문 링크
  - Notion 데이터베이스의 `원문 링크` 속성

- 처리 내용:
  - RSS에서 가져온 기사 원문 링크가 이미 Notion 데이터베이스에 저장되어 있는지 검색한다.
  - 중복 판단 기준은 기사 제목이 아니라 `원문 링크`이다.
  - 같은 제목의 기사가 수정되거나 비슷한 제목의 다른 기사가 존재할 수 있으므로, 가장 고유한 값인 원문 링크를 중복 방지 키로 사용하였다.

- 중복 확인 기준:

```text
Notion 원문 링크 = RSS URL
```

- 출력값:
  - 동일한 원문 링크가 존재하는지 여부
  - 검색 결과 개수

- 다음 단계와의 연결:
  - 검색 결과가 없을 경우에만 HTTP 원문 수집 단계로 이동한다.
  - 이미 같은 링크가 존재하면 중복 기사로 판단하고 워크플로우를 종료한다.

---

### 4-4. 중복 없음 필터 단계

- 사용 모듈: Make 필터
- 입력값:
  - Notion Search Objects 결과 개수

- 처리 내용:
  - Notion 검색 결과가 0개인지 확인한다.
  - 검색 결과가 0개라는 것은 아직 Notion에 저장되지 않은 신규 기사라는 의미이다.
  - 신규 기사만 후속 단계로 넘겨 불필요한 HTTP 요청과 Gemini API 호출을 줄인다.

- 필터 조건:

```text
Total number of bundles = 0
```

- 출력값:
  - 중복이 아닌 신규 기사 데이터

- 다음 단계와의 연결:
  - 중복이 아닌 기사만 HTTP 모듈로 전달되어 원문 HTML을 수집한다.

---

### 4-5. HTTP 원문 HTML 수집 단계

- 사용 모듈: `HTTP - Make a request`
- 입력값:
  - RSS 원문 링크

- 처리 내용:
  - RSS의 Description만으로는 기사 전체 내용을 충분히 요약하기 어렵기 때문에, 원문 링크에 직접 HTTP GET 요청을 보낸다.
  - HTTP 요청 결과로 기사 페이지의 HTML 전체를 가져온다.
  - 이 HTML에는 기사 본문뿐만 아니라 메뉴, 광고, 추천 기사, 기자 정보, 푸터 등도 포함될 수 있으므로, 이후 단계에서 본문만 추출한다.

- HTTP 설정:

```text
Method: GET
URL: RSS URL
Authentication: 없음
Parse response: No
```

- 출력값:
  - 기사 원문 페이지의 HTML 데이터

- 다음 단계와의 연결:
  - HTTP로 가져온 HTML 데이터는 Text Parser 단계로 전달된다.
  - Text Parser는 HTML 전체에서 기사 본문 문단만 추출한다.

---

### 4-6. Text Parser 본문 문단 추출 단계

- 사용 모듈: `Text Parser - Match pattern`
- 입력값:
  - HTTP 모듈에서 가져온 HTML 데이터

- 처리 내용:
  - 기사 페이지 HTML에서 `<p>` 태그에 포함된 문단만 추출한다.
  - AI타임스 기사 본문은 주로 `<p>` 태그 안에 들어 있으므로, HTML 전체를 Gemini에 전달하지 않고 본문에 가까운 문단만 선별한다.
  - 이를 통해 광고, 메뉴, 관련 기사 등 불필요한 텍스트가 요약에 섞이는 것을 줄인다.

- 사용한 정규식:

```regex
<p[^>]*>[\s\S]*?<\/p>
```

- 출력값:
  - `<p>...</p>` 형태로 추출된 여러 개의 문단

- 다음 단계와의 연결:
  - 추출된 문단들은 여러 bundle로 나뉘어 출력되므로, Text Aggregator로 전달하여 하나의 본문으로 병합한다.

---

### 4-7. Text Aggregator 본문 병합 단계

- 사용 모듈: `Tools - Text aggregator`
- 입력값:
  - Text Parser에서 추출된 여러 개의 `<p>` 문단

- 처리 내용:
  - Text Parser가 추출한 여러 문단을 하나의 긴 본문 텍스트로 합친다.
  - 문단 사이에는 줄바꿈을 넣어 기사 본문의 구조를 최대한 유지한다.
  - 이 단계를 거치지 않으면 Gemini가 문단 단위로 여러 번 실행될 수 있으므로, 반드시 하나의 본문으로 병합해야 한다.

- 출력값:
  - 하나로 합쳐진 기사 본문 HTML 조각

- 다음 단계와의 연결:
  - 병합된 본문은 HTML to text 단계로 전달되어 태그와 HTML 엔티티를 정리한다.

---

### 4-8. HTML to text 본문 정제 단계

- 사용 모듈: `Text Parser - HTML to text`
- 입력값:
  - Text Aggregator에서 병합된 본문 HTML

- 처리 내용:
  - `<p>`, `<a>`, `<strong>` 등의 HTML 태그를 제거한다.
  - `&nbsp;`, `&#39;`, `&middot;` 같은 HTML 엔티티를 사람이 읽기 쉬운 텍스트로 변환한다.
  - Gemini에 전달하기 전 본문을 최대한 깔끔한 텍스트 형태로 정리한다.

- 출력값:
  - 정제된 기사 본문 텍스트

- 다음 단계와의 연결:
  - 정제된 본문은 Gemini AI 요약 단계의 입력값으로 사용된다.

---

### 4-9. Gemini AI 요약 단계

- 사용 모듈: `Google Gemini AI - Generate a response`
- 입력값:
  - RSS 기사 제목
  - RSS 원문 링크
  - HTML to text로 정제된 기사 본문

- 처리 내용:
  - 정제된 기사 본문을 바탕으로 한국어 요약문을 생성한다.
  - 요약 결과는 Notion의 `요약문` 속성에 바로 저장할 수 있도록 3줄 이내의 bullet 형식으로 출력하도록 지시하였다.

- Gemini 프롬프트 조건:

```text
다음 뉴스 기사 본문을 한국어로 3줄 이내로 요약해줘.

조건:
- 반드시 3줄 이내로 작성
- 각 줄은 "- "로 시작
- 기사에 없는 내용은 추측하지 않기
- 로봇 기술 동향, 기업/시장 영향, 핵심 시사점을 중심으로 요약
- 광고, 메뉴, 댓글, 관련기사, 기자 정보는 요약하지 않기
- 요약문만 출력하고 "요약:" 같은 제목은 붙이지 않기
```

- 출력값:
  - Gemini가 생성한 3줄 이내 한국어 요약문

- 다음 단계와의 연결:
  - Gemini 요약 결과는 Notion 저장 단계로 전달된다.

---

### 4-10. Notion 데이터베이스 저장 단계

- 사용 모듈: `Notion - Create a Database Item`
- 입력값:
  - RSS 기사 제목
  - Gemini 요약 결과
  - RSS 원문 링크
  - RSS 발행일시

- 처리 내용:
  - 최종 처리된 기사 정보를 Notion 데이터베이스에 저장한다.
  - 정상 처리된 기사는 `상태` 값을 `성공`으로 저장한다.
  - 원문 링크는 추후 중복 확인을 위한 기준 값으로 다시 사용된다.

- Notion 저장 속성:

| 속성명 | 타입 | 저장 값 |
|---|---|---|
| 제목 | Title | RSS 기사 제목 |
| 요약문 | Rich text | Gemini 요약 결과 |
| 원문 링크 | URL | RSS 원문 링크 |
| 발행일시 | Date | RSS pubdate |
| 상태 | Select | 성공 |

- 출력값:
  - Notion 데이터베이스에 저장된 뉴스 요약 항목

- 다음 단계와의 연결:
  - 정상 처리 흐름은 이 단계에서 종료된다.
  - 이후 같은 기사가 RSS에서 다시 감지되더라도, 원문 링크 중복 확인 단계에서 걸러진다.

## 5. 오류 처리 방식

이 개인 워크플로우는 별도의 오류 모니터링 시나리오를 만들기 전 단계의 구조를 기준으로 한다. 따라서 오류 처리는 메인 시나리오 내부에서 Make가 제공하는 기능을 중심으로 구성하였다.

### 5-1. 초기 오류 처리 계획

초기에는 HTTP 요청 또는 Gemini 요약 과정에서 오류가 발생하면, 오류 내용을 Notion에 저장하고 다시 재시도하는 방식을 고려하였다.

그러나 구현 과정에서 오류 발생 즉시 Notion에 로그를 남긴 뒤 Retry를 수행하면, 이후 재시도에 성공하더라도 실패 로그가 남는 문제가 있었다. 또한 Make의 Retry 기능은 실패한 실행을 다시 시도하는 데에는 적합하지만, 재시도 성공 여부에 따라 이후 흐름을 세밀하게 분기하는 데에는 제약이 있었다.

따라서 별도 오류 모니터링 시나리오를 만들기 전 단계에서는 Make의 기본 오류 관리 기능을 활용하는 방식으로 정리하였다.

### 5-2. 최종 적용한 오류 처리 방식

- HTTP 요청 또는 Gemini 요약 단계에서 오류가 발생하면 Make의 Retry 기능을 통해 최대 2회 자동 재시도한다.
- Retry 이후에도 실패한 실행은 Make의 Incomplete executions에 저장되도록 설정한다.
- 오류 발생 내역은 Make의 History에서 확인할 수 있다.
- 별도의 Notion Error Log 데이터베이스는 이 단계에서는 사용하지 않는다.

오류 처리 흐름은 다음과 같다.

```text
HTTP 또는 Gemini 오류 발생
→ Make Retry 기능으로 최대 2회 재시도
→ 성공하면 이후 흐름 계속 진행
→ 계속 실패하면 Incomplete executions에 저장
→ Make History에서 오류 원인과 실행 흐름 확인
```

### 5-3. 선택 이유

이 방식을 선택한 이유는 다음과 같다.

- Make가 자체적으로 실행 기록과 오류 메시지를 History에 저장한다.
- Retry 기능을 통해 일시적인 네트워크 오류나 API 오류를 자동으로 복구할 수 있다.
- 오류 로그를 Notion에 먼저 남기고 재시도하는 방식은 재시도 성공 시 불필요한 오류 기록이 남을 수 있다.
- 별도의 오류 로그 DB를 만들지 않아도 Make History에서 오류 발생 모듈, 오류 메시지, 실행 흐름을 확인할 수 있다.
- 구조가 단순하여 개인 워크플로우 단계에서는 유지보수가 쉽다.

## 6. Notion 저장 결과

정상적으로 처리된 기사는 Notion 데이터베이스에 다음 정보로 저장된다.

| 항목 | 내용 |
|---|---|
| 제목 | RSS에서 가져온 기사 제목 |
| 요약문 | Gemini가 생성한 3줄 이내 요약문 |
| 원문 링크 | 기사 원문 URL |
| 발행일시 | RSS pubdate |
| 상태 | 성공 |

## 8. 특징 및 개선점

### 8-1. 특징

- RSS에서 모든 뉴스를 가져오지 않고 `로봇` 관련 기사만 선별한다.
- Notion의 `원문 링크`를 기준으로 중복 저장을 방지한다.
- Gemini 호출 이전에 중복 확인을 수행하여 불필요한 AI API 호출을 줄인다.
- HTML 전체가 아니라 `<p>` 태그 기반으로 본문을 추출해 요약 품질을 높인다.
- Text Aggregator와 HTML to text를 사용해 Gemini 입력을 정제한다.
- 오류 처리는 Make의 Retry와 History, Incomplete executions를 활용하여 단순하게 구성하였다.

### 8-2. 개선 가능성

- 별도의 오류 모니터링 시나리오를 추가하면 Incomplete executions에 남은 실패 실행을 Notion에 자동 기록할 수 있다.
- 기사 사이트 구조가 바뀌면 `<p>` 태그 기반 본문 추출 방식도 수정이 필요할 수 있다.
- 필터 키워드를 확장하면 로봇 외에도 반도체, 생성형 AI, 정책 등 다른 분야의 뉴스 자동화로 확장할 수 있다.
- Gemini 요약 프롬프트를 조정하면 시장 영향 중심, 기술 동향 중심, 투자 관점 중심 등 다양한 요약 형식으로 변경할 수 있다.
