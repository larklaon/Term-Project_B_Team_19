# 워크플로우 구조

## 1. 전체 구조

본 워크플로우는 수집, 선별, 중복 확인, 본문 가공, AI 요약, 저장의 순서로 구성된다.

```text
RSS Watch Feed Items
→ 로봇 기사 필터
→ Notion Search Objects
→ 중복 없음 필터
→ HTTP Make a request
→ Text Parser Match Pattern
→ Text Aggregator
→ HTML to text
→ Gemini Generate a response
→ Notion Create Database Item
```

## 2. 모듈별 필요성

### 2.1 RSS Watch Feed Items

AI타임스 RSS 피드를 정해진 주기에 확인하고 제목, 설명, 요약, 원문 링크, 발행일시를 가져온다. 웹 페이지를 직접 순회하지 않고 표준 RSS 데이터를 이용하므로 수집 구조가 단순하고 안정적이다.

### 2.2 로봇 기사 필터

제목, 설명 또는 요약에 `로봇`이 포함된 기사만 통과시킨다. 전체 AI 뉴스 가운데 프로젝트 주제와 관련된 기사로 처리 범위를 좁혀 불필요한 후속 요청을 줄인다.

### 2.3 Notion Search Objects

RSS에서 받은 원문 링크와 동일한 값이 Notion 데이터베이스에 존재하는지 검색한다. 원문 링크를 중복 방지 키로 사용하여 이미 저장된 기사가 다시 처리되는 것을 막는다.

### 2.4 중복 없음 필터

Notion 검색 결과가 없는 경우에만 다음 단계로 전달한다. 중복 기사는 HTTP와 Gemini 모듈에 도달하지 않으므로 처리 시간과 API 호출 비용이 절감된다.

### 2.5 HTTP Make a request

RSS 설명문만으로 부족한 기사 정보를 보완하기 위해 원문 링크로 GET 요청을 보내 페이지 HTML을 가져온다. 이 결과는 본문 추출 단계의 입력으로 사용된다.

### 2.6 Text Parser Match Pattern

HTML 전체에서 `<p>` 태그에 해당하는 문단을 정규식으로 추출한다. 메뉴, 광고, 푸터 등 불필요한 페이지 요소를 줄이고 기사 본문에 가까운 텍스트만 후속 단계로 전달하기 위해 필요하다.

### 2.7 Text Aggregator

Text Parser가 여러 결과로 출력한 문단을 하나의 기사 본문으로 병합한다. 이를 통해 문단마다 Gemini 호출이 발생하지 않도록 하고 기사 1건당 1회의 요약 호출 원칙을 지킨다.

### 2.8 HTML to text

병합된 본문의 HTML 태그와 엔티티를 일반 텍스트로 변환한다. Gemini에 불필요한 마크업이 전달되는 것을 줄여 입력 품질과 요약 정확도를 높인다.

### 2.9 Gemini Generate a response

기사 제목, 원문 링크, 정제된 본문을 바탕으로 3줄 이내의 한국어 요약문을 생성한다. 기사에 없는 내용을 추측하지 않고 로봇 기술 동향, 기업·시장 영향, 핵심 시사점에 집중하도록 프롬프트를 적용한다.

### 2.10 Notion Create Database Item

RSS 메타데이터와 Gemini 결과를 Notion 속성에 매핑해 저장한다. 정상 처리된 항목에는 상태를 `성공`으로 기록하며, 원문 링크는 이후 실행의 중복 판단 기준으로 재사용된다.

## 3. 구조 설계의 핵심

- 주제 필터와 중복 확인을 AI 호출 이전에 배치하여 비용을 줄였다.
- 여러 본문 문단을 먼저 병합하여 기사 1건당 요약 호출이 1회만 발생하도록 구성했다.
- 오류 재시도는 최대 2회로 제한하고, 최종 실패는 Incomplete executions에 보관한다.
- 메인 처리와 오류 모니터링을 분리하여 정상 뉴스 데이터와 오류 기록을 안정적으로 관리한다.

전체 연결 구조와 모듈별 설정 화면은 `스크린샷/01_전체_워크플로우/` 및 각 단계별 스크린샷 폴더에서 확인할 수 있다.
