# 에러 처리 정책

## 1. 정책 목표

오류 처리의 목표는 일시적인 네트워크·API 장애를 자동으로 복구하고, 복구되지 않은 실행을 누락 없이 추적하면서도 재시도와 로그 저장으로 인한 불필요한 API 호출을 제한하는 것이다.

## 2. 초기 계획

초기에는 HTTP 요청 또는 AI 요약 과정에서 오류가 발생하면 2회 재시도 후 Notion Error Log에 기록하는 구조를 계획하였다.

이 방식은 하나의 오류 처리 경로에서 재시도와 최종 실패 기록을 모두 수행하려는 설계였다.

## 3. 구현 과정에서 확인한 점

Make의 Error Handler는 Retry 기능을 제공하지만, 재시도 후 성공했을 때의 흐름과 실패 후 별도 저장 흐름을 원하는 방식으로 세밀하게 구성하는 데에는 제약이 있었다.

따라서 메인 시나리오 안에서 모든 재시도 결과와 오류 저장 분기를 복잡하게 연결하기보다, Make가 기본으로 제공하는 실행 이력 관리 기능을 활용하는 편이 안정성과 유지보수 측면에서 적합하다고 판단하였다.

## 4. 최종 방식

HTTP 요청이나 Gemini 요약 단계에서 오류가 발생하면 Make의 Retry 기능을 통해 최대 2회 자동 재시도한다. 재시도 후에도 실패한 실행은 Make의 Incomplete executions에 저장되며, 별도의 오류 모니터링 시나리오가 `Asia/Seoul` 기준 매일 09:10에 이를 조회하여 Notion에 오류 로그로 기록한다.

오류 모니터링 시나리오는 Incomplete execution의 오류 ID를 기준으로 Notion을 먼저 검색한다. 동일 오류 ID가 이미 존재하면 저장을 중단하고, 검색 결과가 없을 때만 신규 오류 로그를 생성한다.

## 5. 오류 처리 흐름

오류 모니터링 시나리오는 메인 뉴스 요약 시나리오가 09:00에 실행된 뒤인 매일 09:10에 실행된다.

```text
HTTP/Gemini 오류 발생
→ Retry 최대 2회
→ 계속 실패하면 Make Incomplete executions에 저장
→ 오류 모니터링 시나리오가 Incomplete executions 조회
→ 오류 ID 기준으로 Notion 중복 확인
→ 중복이 없으면 Notion에 오류 로그 저장
```

## 6. 선택 이유

- Make의 History와 Incomplete executions를 통해 오류 발생 모듈, 오류 메시지, 실행 흐름, 재시도 이력을 확인할 수 있다.
- 일시적인 네트워크 오류나 API 오류는 Retry로 자동 복구를 시도할 수 있다.
- 재시도 후에도 실패한 항목은 Notion에 오류 로그로 기록할 수 있다.
- 오류 ID를 기준으로 중복 로그 저장을 방지할 수 있다.
- 모든 오류를 즉시 Notion에 저장하는 방식보다 불필요한 API 호출을 줄일 수 있다.
- Make의 기본 오류 관리 기능을 활용하므로 유지보수가 간편하다.

## 7. 재시도 및 비용 관리 원칙

- 요약 호출은 기사 1건당 1회를 원칙으로 한다.
- 오류로 인한 재시도만 예외로 허용하며, 최대 2회로 제한한다.
- 주제 필터와 원문 링크 중복 확인을 Gemini 이전에 수행한다.
- 중복 기사나 관련 없는 기사에는 HTTP 요청과 Gemini 호출을 수행하지 않는다.
- 오류 모니터링 시나리오에서도 오류 ID 중복 확인 후에만 Notion 저장 모듈을 실행한다.

## 8. 운영 및 보안 주의사항

운영자는 Make History와 Incomplete executions를 주기적으로 확인하여 반복 오류의 원인을 분석한다. 오류 로그에는 문제 해결에 필요한 정보만 기록하며 API 키, 인증 토큰, Authorization 헤더, 개인 이메일, 비공개 링크 등 민감한 정보는 저장하거나 화면에 노출하지 않는다.

Retry 설정과 오류 모니터링 시나리오 화면은 `스크린샷/12_오류_처리_및_모니터링/`에서 확인할 수 있다.
